Browser vs Node.js

Полный разбор Event Loop — как работает и в чём принципиальное отличие

00 — основа

Что такое Event Loop?

JavaScript — однопоточный язык. Он может выполнять только одну операцию в каждый момент времени. Но браузер и сервер должны обрабатывать сеть, файлы, таймеры — всё одновременно.

Event Loop — это механизм, который решает эту проблему. Он постоянно крутится и решает: что выполнять следующим, когда Call Stack пустой.

Общая модель (работает везде)

📚 Call Stack
Стек вызовов. Сюда попадает весь синхронный код. Выполняется сразу, LIFO.
⚡ Microtask Queue
Очередь микрозадач. Выполняется сразу после каждой задачи, до следующего рендера/тика.
🕐 Macrotask Queue
Очередь задач. setTimeout, setInterval, I/O события. Берётся по одной за раз.

Ключевые моменты

🎨
Браузер рендерит DOM между тиками — это уникальная черта. Event Loop синхронизирован с отрисовкой.
⚡
Микрозадачи выполняются полностью перед рендером. Бесконечная очередь микрозадач = заморозка страницы.
🖱
События DOM (click, input) — это макрозадачи. Каждый колбэк = отдельный тик.
🔧
Web APIs (fetch, setTimeout) — это не JS. Они работают в браузере параллельно, а результат кладут в очередь.

Ключевые моменты

🔄
Node.js использует libuv — C-библиотеку, которая управляет фазами. Это не браузерный API.
🚫
Нет рендера. Нет DOM. Цикл крутится максимально быстро, без привязки к 60fps.
⚡
process.nextTick() — особый приоритет. Выполняется раньше любых Promise, между фазами. Мощный, но опасный инструмент.
📂
Фаза poll — сердце сервера. Здесь Node ждёт новых данных из файлов и сети. Именно тут происходит вся I/O магия.
⚡
setImmediate() всегда выполняется после I/O текущего тика — удобно для отложенного выполнения после I/O.

Browser vs Node.js — таблица отличий

Параметр 🌐 Browser 🟢 Node.js
Движок HTML Living Standard (W3C) libuv (C-библиотека)
Рендер ✅ Между тиками (~60fps) ❌ Нет DOM, нет рендера
Структура Call Stack → Micro → Render → Macro 6 фаз: timers → pending → idle → poll → check → close

04 — пример с порядком выполнения
КОД

console.log('1: sync');

setTimeout(() => console.log('2: timeout'), 0);

Promise.resolve()
  .then(() => console.log('3: promise'));

queueMicrotask(() => console.log('4: microtask'));

if (typeof process !== 'undefined') {
  process.nextTick(() => console.log('5: nextTick'));
}

console.log('6: sync end');

⚡ ГЛАВНОЕ, ЧТО НУЖНО ЗАПОМНИТЬ

1️⃣
Синхронный код всегда первый. Call Stack очищается полностью.
2️⃣
Микрозадачи всегда раньше макрозадач. Promise.then() выполнится до следующего setTimeout.
3️⃣
В Node.js process.nextTick() — самый высокий приоритет среди асинхронных задач. Выполняется до Promise.
4️⃣
Главное отличие браузера — наличие фазы рендера. Event Loop браузера работает в ритме экрана.
5️⃣
Главное отличие Node.js — 6 явных фаз libuv с poll-фазой в центре. Микрозадачи дренируются между каждой фазой, а не только между макрозадачами.
⚠️
Блокировка стека = зависание. Тяжёлые вычисления блокируют весь цикл — и в браузере (интерфейс), и в Node.js (все запросы).
